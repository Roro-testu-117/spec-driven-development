# LLM生成機能の追加ルール

plan.md がLLM/プロンプト/生成/評価に関する機能を含む場合、以下のルールを tasklist.md 作成時に追加適用する。

---

## 追加ルール

1. LLM活用機能では、生成タスクの直後に Judge 評価タスクを配置する
2. Judge 評価の結果形式は `passed: bool` と `reason: str` に統一する
   - 例（Judge評価結果）:
   ```json
   {
     "passed": true,
     "reason": "必須要件A/Bを満たし、禁止事項に抵触しないため"
   }
   ```
   - `passed=false` の場合も `reason` は必須で、基準未達の理由を記録する
3. Judge 評価は固定の評価データセット（golden set）を利用し、評価セットIDを結果に記録する
4. LLM の生成出力・評価出力は Pydantic の厳格スキーマで定義する
5. Pydantic パース失敗時は 1 回だけ再試行し、再失敗した場合は未達として記録する
   - 記録する情報: 失敗理由（validation error要約）、対象入力ID、再試行有無

## LLM向けフェーズ構成例

汎用テンプレートのフェーズ1-3を以下のように置き換える。フェーズ4（品質チェック）はそのまま使用する。

```markdown
## フェーズ1: LLM入出力設計
- [ ] 1. 生成I/O仕様を定義する
  - 目的
    - 入力、出力、制約を明確化して実装の前提を固定する
- [ ] 1.1 入力データモデルを定義する
  - 成果物（ファイル）
    - [ ] src/{機能}/models/input.py
  - Done条件
    - [ ] 入力項目と型が定義され、利用側から参照できる
- [ ] 1.2 出力データモデルを定義する
  - Depends on（任意）
    - 1.1
  - 成果物（ファイル）
    - [ ] src/{機能}/models/output.py
  - Done条件
    - [ ] 出力項目と型が定義され、呼び出し側が期待形式を判定できる

## フェーズ2: 生成機能実装
- [ ] 2. LLM生成処理を実装する
  - 目的
    - ユースケースを満たす生成処理を成立させる
- [ ] 2.1 プロンプトと呼び出し処理を実装する
  - 成果物（ファイル）
    - [ ] src/{機能}/chains/generate.py
    - [ ] src/{機能}/prompts/generate_prompt.txt
  - Done条件
    - [ ] 想定入力に対して期待される出力を返す処理が接続されている
- [ ] 2.2 失敗時ハンドリングを実装する
  - Depends on（任意）
    - 2.1
  - 成果物（ファイル）
    - [ ] src/{機能}/chains/generate.py
  - Done条件
    - [ ] 失敗時の返却形式または例外処理方針がコード上で統一されている

## フェーズ3: 評価と記録
- [ ] 3. 生成結果の評価を実装する
  - 目的
    - 生成品質を同一基準で判定し、改善可能な記録を残す
- [ ] 3.1 評価基準と評価入力を定義する
  - Depends on（任意）
    - 2.1
  - 成果物（ファイル）
    - [ ] src/{機能}/eval/rubric.md
    - [ ] src/{機能}/eval/eval_dataset.jsonl
  - Done条件
    - [ ] 評価基準と評価対象入力が追跡可能な形式で定義されている
- [ ]* 3.2 Judge評価結果の保存処理を実装する
  - Depends on（任意）
    - 3.1
  - 成果物（ファイル）
    - [ ] src/{機能}/eval/run_judge.py
    - [ ] src/{機能}/eval/results/latest.jsonl
  - Done条件
    - [ ] 評価結果が継続的に比較可能な形式で記録される
```
