# コードレビュースキル (spec-code-reviewer)

**コンセプト**: design.md と tasklist.md に対して実装内容を評価し、レビュー記録を残す

**引数**:
- 機能名（必須）: `.steering` 配下ディレクトリ名に使う

**使用例**:
```bash
/spec-code-reviewer rag-ui
```

---

## 参照ドキュメント

| 作業 | 常に開くドキュメント | 参照するドキュメント |
|------|---------------------|---------------------|
| コードレビュー | design.md, tasklist.md | evaluation-criteria.md, 実装コード（git diff） |

**レビュー時は design.md の実装方針と tasklist.md の完了状況を確認しながら、実装コードを評価する**

---

## 手順

1. **対象ディレクトリの決定**
   - `.steering/.steering-{機能名}-{YYYYMMDD}` を特定する。
   - 複数候補がある場合はユーザーに確認する。

2. **入力ドキュメントの確認**
   - `design.md` と `tasklist.md` を読み込む。
   - 不足があればユーザーに作成を依頼する。

3. **コードレビューの実施**
   - 評価軸は以下を参照:
     ```
     Read('.claude/skills/spec-code-reviewer/templates/evaluation-criteria.md')
     ```
   - 各観点を ✓ / △ / ✗ で評価し、コメントを付与する。

4. **code-reviewed.md への記録**
   - レビュー結果を `code-reviewed.md` に追記する。
   - 「第N回レビュー（YYYY-MM-DD HH:MM）」の形式で追記する。
   - テンプレートは以下を参照:
     ```
     Read('.claude/skills/spec-code-reviewer/templates/code-reviewed.md')
     ```

5. **結果報告**
   - 全項目が ✓ または △ の場合は「OK」と判定する。
   - ✗ がある場合は「修正後に再レビュー」と判定する。
   - セキュリティが ✗ の場合は「即座に修正必須」と強調する。

---

## 修正ループ

✗ がある場合の処理:

1. **修正依頼の記録**
   - `code-reviewed.md` の「指摘事項」に修正が必要な箇所を具体的に記載する。
   - ファイルパスと行番号を明記する: `{ファイル}:{行} - {指摘内容}`

2. **ユーザーへの報告**
   ```
   コードレビュー完了: 修正が必要です。

   【修正が必要な箇所】
   - {ファイル}:{行} - {指摘内容}

   修正後、再度 /spec-code-reviewer {機能名} を実行してください。
   ```

3. **セキュリティ問題の場合**
   - セキュリティが ✗ の場合は、他の修正より優先して即座に修正を依頼する。
   ```
   ⚠️ セキュリティ問題が検出されました。即座に修正してください。
   - {ファイル}:{行} - {セキュリティ問題の内容}
   ```

4. **再レビュー**
   - ユーザーが修正後に `/spec-code-reviewer` を再実行する。
   - 「第N回レビュー」として追記形式で記録する。

---

## 最終完了

全項目が ✓ または △ の場合:

1. **code-reviewed.md の更新**
   - 「総合判定」を「OK」にチェックする。

2. **ユーザーへの報告**
   ```
   コードレビュー完了: OK

   【レビュー結果サマリー】
   - 設計適合性: ✓
   - タスク完了度: ✓
   - コード品質: ✓
   - セキュリティ: ✓
   - テスト: ✓

   実装が完了しました。お疲れ様でした！
   ```

3. **振り返りの提案**（任意）
   - `tasklist.md` の「振り返り」セクションへの記入を提案する。
