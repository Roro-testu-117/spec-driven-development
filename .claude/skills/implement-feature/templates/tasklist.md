# タスクリスト

## 運用ルール（3-agent loop）

**このセクションは削除・改変しないこと**

### CT（Commit Task）駆動の実装フロー

```
┌─────────────────┐
│ task-decomposer │ ← tasklist.md を CT に分解
└────────┬────────┘
         │ CT定義
         ▼
┌─────────────────┐
│  task-executor  │ ← 1 CT = 1コミットで実装
└────────┬────────┘
         │ 実装完了
         ▼
┌─────────────────┐
│ quality-checker │ ← レビュー & 品質チェック
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
   OK        NG
    │         │
    ▼         ▼
 コミット   修正 or 差し戻し
```

### 呼び出しルール（必須）

1. **作業開始前**: task-decomposer により CT を確定し、`task-decomposed/` に記述
2. **各CT実装後**: quality-checker によるレビュー
3. **OK の場合のみ**: task-executor がコミットメッセージを提案 → **ユーザーがコミット実行**
4. **Blocked / 差し戻し時**: task-decomposer に戻して CT を再分解・調整

### コンテキスト・パケット（呼び出し時に必ず渡す）

- `feature-docs/requirements.md`
- `feature-docs/design.md`
- `feature-docs/task-decomposed/ct-xxx.md`
- 変更対象ファイル一覧
- 参照すべき設計資料の該当箇所（必要最小限）

## タスク完全完了の原則

**このファイルの全タスクが完了するまで作業を継続すること**

### 必須ルール
- **全てのタスクを `[x]` にすること**
- 「時間の都合により別タスクとして実施予定」は禁止
- 「実装が複雑すぎるため後回し」は禁止
- 未完了タスク（`[ ]`）を残したまま作業を終了しない

### タスクスキップが許可される唯一のケース
以下の技術的理由に該当する場合のみスキップ可能:
- 実装方針の変更により、機能自体が不要になった
- 依存関係の変更により、タスクが実行不可能になった

スキップ時は必ず理由を明記:
```markdown
- [x] ~~タスク名~~（実装方針変更により不要: 具体的な技術的理由）
```

### タスクが大きすぎる場合
- タスクを小さなサブタスクに分割
- 分割したサブタスクをこのファイルに追加
- サブタスクを1つずつ完了させる
