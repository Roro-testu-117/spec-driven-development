# CT実装ロジック（統合版）

このファイルは、スキル内の実装ロジックを記述したものです。

## 概要

`feature-docs/task-decomposed/ct-xxx.md` を1つずつ実装し、1コミット単位で作業を完結させます。

## 責務

- `feature-docs/task-decomposed/ct-xxx.md` を1つずつ実装
- **1 CT = 1コミット**
- 実装後、CTファイル内のステータスを更新

## 入力

- `feature-docs/task-decomposed/ct-xxx.md`
- 変更対象ファイル一覧（予定 or 実績）
- 参照すべき設計資料（`feature-docs/design.md` の該当箇所）

## 制約

- **スコープはCTに記載された範囲に限定**
- スコープ拡大が必要な場合は実装を止めて差し戻す
- 依存追加・設定変更はCTに明示がない限り行わない
- 実装中に発見した問題は記録し、必要に応じて新CTの作成を提案

## LangChain/機械学習開発における特記事項

- **APIキー管理**: `.env` ファイルにAPIキーを記載し、`python-dotenv` で読み込む
- **ストリーミング処理**: LangChainのストリーミングAPIを適切に使用する
- **プロンプト管理**: プロンプトテンプレートは変数化し、ハードコードを避ける
- **ベクトルDB操作**: Chroma等のベクトルDBは永続化設定に注意
- **モデル選択**: 用途に応じて適切なLLMモデルを選択（GPT-4, Claude, Gemini等）
- **エラーハンドリング**: API呼び出しのリトライ・タイムアウト処理を実装

## 作業フロー（呼び出しモード別）

### モードA: 初回実装（CTステータス: 未着手 または 差し戻し）

1. **CT読み込み**
   ```
   Read('feature-docs/task-decomposed/ct-XXX.md')
   ```
   - ステータスを「作業中」に更新

2. **コンテキスト確認**
   - 設計資料の該当箇所を確認
   ```
   Read('feature-docs/design.md')
   Read('feature-docs/context.md')
   ```
   - 変更対象ファイルの現状を確認

3. **実装**
   - CTに記載された内容のみを実装
   - スコープ外の変更は行わない
   - Write, Edit, Bash ツールを使用

4. **自己チェック**
   - 完了条件を確認
   - 構文エラー・型エラーがないか確認
   - 基本的な動作確認

5. **ステータス更新**
   - CTファイルのステータスを「レビュー待ち」に更新
   - 実装内容のサマリーをCTファイルに追記

6. **完了報告**
   - 実装完了を報告
   - レビュー実施を促す

---

### モードB: コミット実行準備（レビューOK後）

1. **CT読み込み**
   - CTファイルを読み込み、ステータスが「レビュー待ち」であることを確認

2. **コミット実行準備**
   - コミットメッセージ案を1行で記入（必要な場合のみ）
   - ユーザーに `/create-commit` スキルでのコミット実行を依頼

3. **ステータス更新**
   - CTファイルのステータスを「完了」に更新

4. **tasklist.md 更新**
   - tasklist.md の該当タスクに `[x]` を付ける（CT完了時）

5. **完了報告**
   - コミット実行を依頼済みであることを明記

---

### モードC: 修正（レビューNG判定後、1〜2回目）

1. **CT読み込み**
   - CTファイルを読み込む
   - NG回数を確認（1〜2回目であることを確認）

2. **NG回数更新**
   - CTファイルのNG回数を更新（+1）
   - ステータスを「作業中」に変更

3. **修正実装**
   - レビュー指摘に基づき修正
   - スコープを守り、指摘された問題のみを修正

4. **自己チェック**
   - 修正内容の確認
   - 構文エラー・型エラーがないか確認

5. **ステータス更新**
   - CTファイルのステータスを「レビュー待ち」に更新
   - 実装サマリーを更新

6. **完了報告**
   - 修正完了を報告
   - レビュー再実施を促す

---

### モードD: 差し戻し処理（レビューNG判定後、3回目以降）

1. **CT読み込み**
   - CTファイルを読み込む
   - NG回数を確認（3回以上であることを確認）

2. **差し戻し処理**
   - CTファイルのステータスを「差し戻し」に変更
   - 差し戻し理由をCTファイルに記録
   - 発生した問題の詳細を記録

3. **完了報告**
   - 差し戻しを報告
   - CT再分解を提案

## スコープ制約の厳守

以下の場合は実装を停止し、差し戻しを行います：

- CTに記載されていない機能追加が必要になった
- 新しい依存パッケージの追加が必要になった
- データ構造の変更が必要になった
- 影響範囲がCTの想定を超えた

**差し戻し時の対応：**
1. 実装を停止
2. CTファイルに問題を記録
3. CT再分解を実施
4. ステータスを「差し戻し」に更新

## レビュー前チェックリスト

- [ ] CTの完了条件をすべて満たしている
- [ ] 構文エラー・型エラーがない
- [ ] スコープ外の変更を含んでいない
- [ ] CTファイルのステータスが「レビュー待ち」に更新されている

## 自己チェック項目（Python/LangChain特有）

- [ ] `.env.example` にAPIキーの例を記載しているか
- [ ] requirements.txt に新しい依存が追加されている場合、記載したか
- [ ] LangChain の chain は適切に構成されているか
- [ ] プロンプトテンプレートに未定義変数がないか
- [ ] ベクトルDB の永続化パスは正しいか
- [ ] エラーハンドリング（API呼び出しのリトライ等）は実装されているか

## コミットメッセージフォーマット（レビューOK後に参考として提示）

```
<type>: task description
```

**重要**: コミット実行はユーザーが `/create-commit` スキルで行う。スキルはメッセージ案を提示するのみ。

**使用するコミットタイプ（以下の10種類のみ）**

| タイプ | 説明 | 使用例 |
|--------|------|--------|
| **feat** | 新しい価値を追加 | 新機能、新規ファイル追加 |
| **fix** | 既存機能の不具合 | バグ修正 |
| **refactor** | 動作は変えず内部改善 | コードの整理、構造の改善 |
| **update** | 既存機能の仕様変更 | 機能の修正、仕様変更 |
| **docs** | READMEやコメント修正 | ドキュメント更新 |
| **test** | テストの追加・修正 | テストコード追加、修正 |
| **chore** | 依存関係、設定ファイル | ライブラリ更新、ビルド設定 |
| **remove** | 不要コードの削除 | ファイル削除、機能削除 |
| **rename** | ファイル名の変更 | ファイル名のリネーム |
| **move** | ファイルの移動 | ディレクトリ構成の変更 |

**重要**: コミットタイプは必ず上記10種類のいずれかを使用すること。

## レビューNG時の対応

### 修正ループ制限
- **1〜2回目のNG**: 修正して再度レビューを実施
- **3回目以降のNG**: CTを再分解（定義を見直し）

### NG回数のカウント方法
- 同一CTに対するレビューNG判定回数をカウント
- CTファイルに「NG回数」を記録
- 3回目のNGで自動的にステータスを「差し戻し」に更新

## 注意事項

- 1 CT = 1コミットを厳守（実装単位として）
- スコープを守り、余計な変更をしない
- レビューを必ず実施する
- NG回数制限を超えた場合は、無理に修正せずCTを再分解する
- **コミット実行はユーザーが行う**。エージェントはコミットメッセージを提案するのみ
