# 品質レビューロジック（統合版）

このファイルは、スキル内の品質レビューロジックを記述したものです。

## 概要

各Commit Task（CT）のコミット前にレビューを行い、設計・仕様・品質観点での確認を実施します。

## 責務

- 各CTのコミット前レビューゲート
- 設計・仕様・品質観点での確認
- コミット可否の判定

## 入力

- `git diff`
- `feature-docs/task-decomposed/ct-xxx.md`
- 設計根拠（`feature-docs/design.md` の該当箇所）

## 出力

- **結論**（OK / NG）
- **重大度**（Blocker / Major / Minor）※NG時のみ
- **根拠**（ファイル・差分・行）
- **修正提案** / 追加テスト観点
- **NG回数**（このCTで何回目のNGか）

## 禁止事項

- 実装を行わない
- ファイル変更を行わない

## レビュー観点

### 1. CT適合性チェック
- [ ] CTに記載された内容のみを実装しているか
- [ ] スコープ外の変更が含まれていないか
- [ ] 完了条件をすべて満たしているか

### 2. 設計適合性チェック
- [ ] 設計資料に従っているか
- [ ] アーキテクチャ原則に違反していないか
- [ ] 命名規則・コーディング規約に従っているか（PEP 8等）

### 3. 品質チェック（Python/LangChain共通）
- [ ] 構文エラー・型エラーがないか
- [ ] テストが必要な場合、適切にテストされているか
- [ ] セキュリティ上の問題がないか（APIキー漏洩、SQLインジェクション等）

### 4. LangChain/生成AI特有のチェック
- [ ] APIキーが `.env` で管理され、コードにハードコードされていないか
- [ ] プロンプトテンプレートの変数は適切に定義されているか
- [ ] LLM呼び出しにタイムアウト・リトライ処理が実装されているか
- [ ] ベクトルDBの永続化設定は正しいか
- [ ] モデル選択は用途に適しているか（コスト・性能バランス）
- [ ] ストリーミング処理は適切に実装されているか

### 5. 機械学習特有のチェック（該当する場合）
- [ ] データ前処理のロジックは適切か
- [ ] モデルの評価指標は適切か
- [ ] 学習データとテストデータは適切に分離されているか
- [ ] ハイパーパラメータは適切に設定されているか

### 6. 影響範囲チェック
- [ ] 変更による影響範囲は適切か
- [ ] 意図しない副作用がないか
- [ ] 後続CTに悪影響を与えないか

## 作業フロー

1. **コンテキスト読み込み**
   ```
   Read('feature-docs/task-decomposed/ct-XXX.md')
   Read('feature-docs/design.md')
   ```

2. **差分確認**
   ```
   Bash('git diff')
   ```
   - 変更されたファイル・行を特定

3. **レビュー実行**
   - 各レビュー観点に基づいてチェック
   - 問題があれば具体的に指摘

4. **品質チェック実行**
   - コードフォーマット: `ruff check .`
   - 型チェック: `mypy .`（型ヒントがある場合）
   - テスト: `pytest`（該当する場合）
   - セキュリティチェック: `.env` ファイルにAPIキーが記載され、コードにハードコードされていないか確認

5. **判定**
   - **OK**: コミット可能
   - **NG**: 修正が必要（具体的な修正提案を提示）

6. **結果報告**
   - 結論と根拠を明確に記載
   - NGの場合は修正方法を具体的に提示

## 出力フォーマット

### OK の場合
```markdown
## レビュー結果: OK

### 確認内容
- CT適合性: ✓
- 設計適合性: ✓
- 品質: ✓
- 影響範囲: ✓

### コメント
[良い点や補足事項]

コミット可能です。
```

### NG の場合
```markdown
## レビュー結果: NG

### NG回数
このCTに対して N 回目のNG判定

### 問題点
1. [問題1]
   - ファイル: xxx.ts:10-15
   - 内容: [具体的な問題]
   - 根拠: [設計資料の該当箇所]
   - 重大度: Blocker / Major / Minor

2. [問題2]
   - ファイル: yyy.ts:20
   - 内容: [具体的な問題]
   - 重大度: Blocker / Major / Minor

### 修正提案
1. [具体的な修正方法1]
2. [具体的な修正方法2]

### 次のアクション
- N < 3: 修正後、再度レビューを実施
- N >= 3: CTを差し戻し（CT定義を見直し）
```

## 品質チェックコマンド失敗時の扱い

以下のコマンドが失敗した場合、自動的にNG判定とします：

- `ruff check .` 失敗 → 自動NG（Major）
  - コードフォーマットの問題
- `mypy .` 失敗 → 自動NG（Major）
  - 型ヒントの問題（型ヒントがある場合のみ）
- `pytest` 失敗 → 自動NG（Blocker）
  - テストの失敗
- APIキーがコードにハードコードされている → 自動NG（Blocker）
  - セキュリティ上の重大な問題

上記以外の観点（設計適合性、影響範囲など）はレビュアー判断とします。

## Python/LangChain特有の注意事項

- `.env` ファイルにAPIキーを記載し、`python-dotenv` で読み込んでいるか
- `requirements.txt` に新しい依存が追加されている場合、記載されているか
- LangChainの chain 構成は適切か
- プロンプトテンプレートに未定義変数がないか
- ベクトルDBの永続化パスは正しいか（絶対パス/相対パスの確認）

## 注意事項

- レビューは客観的な基準に基づいて実施
- 問題を指摘する際は具体的に（ファイル・行・理由）
- 修正提案は実行可能な形で提示
- OKの場合も改善点があれば提案する
- 実装は行わず、レビューに専念する
- NG回数を必ず記録し、3回目以降は差し戻しを推奨
