# タスク分解ロジック（統合版）

このファイルは、スキル内のタスク分解ロジックを記述したものです。

## 概要

`feature-docs/tasklist.md` を「CT（Commit Task）」単位に分解し、実行可能な計画を作成します。

## 責務

- `feature-docs/tasklist.md` を「CT（Commit Task）」単位に分解
- 実行順序・依存関係の確定
- スコープ過大なタスクの分割提案

## 入力

- `feature-docs/requirements.md`
- `feature-docs/design.md`
- `feature-docs/tasklist.md`

## 出力

- `feature-docs/task-decomposed/` 配下の CT ファイル
  - ファイル名: `ct-001.md`, `ct-002.md`, ...
  - 各CTファイルには以下を含める:
    - タスク名
    - ステータス（未着手 / 作業中 / レビュー待ち / 差し戻し / 完了）
    - 目的
    - 対象範囲（変更対象ファイル / ディレクトリ）
    - 実装手順
    - 依存関係
    - 完了条件（テスト / コマンド / 期待挙動）
    - 実行コマンド（必要な場合のみ）
    - 参照（設計資料の該当箇所）

## 禁止事項

- 実装を行わない
- CT定義ファイル以外のファイル変更を行わない

## 作業フロー（モード別）

### モードA: 初回CT分解

1. **コンテキスト読み込み**
   ```
   Read('feature-docs/requirements.md')
   Read('feature-docs/design.md')
   Read('feature-docs/tasklist.md')
   ```

2. **タスク分析**
   - 各タスクの複雑度を評価
   - スコープが大きすぎる場合は分割提案
   - 依存関係の洗い出し

3. **CT分解**
   - 1コミット単位で完結する粒度に分解
   - 依存関係を明確化
   - 実行順序を決定

4. **CT定義ファイル作成**
   - CTテンプレートを読み込む:
     ```
     Read('.codex/skills/implement-feature/templates/ct-template.md')
     ```
   - `feature-docs/task-decomposed/ct-XXX.md` を作成
   - 各CTに明確な完了条件を記載

5. **完了報告**
   - 作成したCT一覧を提示
   - 実行順序と依存関係を説明

---

### モードB: CT再分解（差し戻し時）

1. **差し戻しCT確認**
   - 差し戻されたCTファイルを読み込む
   - 差し戻し理由を確認
   - NG回数とレビュー指摘を確認

2. **問題分析**
   - なぜ差し戻されたかを分析
   - スコープが大きすぎるか
   - 依存関係に問題があるか
   - 実装手順が不明確か

3. **再分解戦略の決定**
   - **スコープ過大**: より小さな複数のCTに分割
   - **依存関係不明確**: 依存CTを新規作成
   - **手順不明確**: 実装手順を詳細化

4. **新CT作成**
   - 元のCTファイルを削除または「差し戻し済み」として保存
   - 新しいCTファイル（ct-XXX-v2.md など）を作成
   - 元のCTの問題を解決した内容にする

5. **完了報告**
   - 再分解結果を提示
   - 元のCTとの違いを説明
   - 新しい実行順序を提示

## CTファイルのフォーマット

```markdown
# CT-XXX: [タスク名]

## ステータス
- [ ] 未着手
- [ ] 作業中
- [ ] レビュー待ち
- [ ] 差し戻し
- [ ] 完了

## レビュー履歴
- NG回数: 0回

## 目的
[このCTで達成すること（1行）]

## 対象範囲
- 変更対象ファイル1
- 変更対象ファイル2
- 変更対象ディレクトリ/

## 実装手順
1. [手順1]
2. [手順2]
3. [手順3]

## 依存関係
- 前提: CT-XXX
- 後続: CT-XXX

## 完了条件
- [ ] 条件1（テスト / コマンド / 期待挙動）
- [ ] 条件2

## 実行コマンド
（必要な場合のみ記載）
```bash
# 例: Chainの手動テスト実行
python -m src.chains.qa_chain

# 例: LLM呼び出しテスト
python scripts/test_llm.py

# 例: ベクトルDB初期化
python scripts/init_vectorstore.py

# 例: テスト実行
pytest tests/test_chain.py -v
```

## 参照
- 設計資料: feature-docs/design.md:XX-XX
- 要件: feature-docs/requirements.md:XX-XX
（具体的なファイルパスと行番号を記載すること）
```

## CT粒度の判断基準

CTは以下の基準を満たす粒度に分解してください：

- **変更ファイル数**: 1〜5ファイル程度
- **実装手順**: 3〜7項目程度
- **推定作業時間**: 30分〜2時間程度（AI実装基準）
- **1コミット単位**: 独立してテスト・レビュー・ロールバック可能な単位

これらの基準を超える場合は、CTを分割してください。

## 注意事項

- CTは1コミット単位で完結する粒度にする
- 各CTは独立してテスト・レビュー可能にする
- スコープが大きい場合は積極的に分割する
- 依存関係を明確にし、並列実行可能なCTは明示する
- 依存関係の循環を検出した場合は警告する
